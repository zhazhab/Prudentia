# Prudentia Architecture

[中文](architecture.md)

## Shape

Prudentia is a local-first monorepo:

- `backend`: Rust API service using Axum, Tokio, SQLx, and SQLite.
- `frontend`: React + Vite + TypeScript workbench.
- `docs`: product and implementation notes.
- `examples`: import templates and sample data.

The backend owns persistence and all provider integrations. The frontend reads files in the browser, sends import content to the backend for preview/commit, and renders the currently wired chat-first memo home, portfolio, memo, and AI settings workflows.

Engineering style is documented in [engineering-style.en.md](engineering-style.en.md). Readability, maintainability, and explainability are treated as architecture constraints, not cosmetic preferences.

## Backend Modules

- `memo`: thesis notes, risks, catalysts, disconfirming evidence, tags, and memo AI extraction.
- `conversation`: the deep `ConversationEngine` module that owns subject resolution, progressive context, external research, real model calls, task lifecycle, durable events, and structured action proposals.
- `memo_thread`: low-level thread/message persistence, pagination, archive, and soft delete.
- `research`: local research records, article/person thought distillation, stock snapshots, portfolio reviews, and adoption of candidate principles/checklist items.
- `investment_system`: versioned executable DAGs, the fixed-rule kernel, skill/agent adapters, JSON Schema validation, and execution traces; the old natural-language system remains migration material only.
- `portfolio`: import preview, mapping, local security symbol directory, holding baselines, immutable trade/reversal ledger, position projections, TWR, summaries, and refresh orchestration.
- `market_data`: quote/FX provider trait with mock, Yahoo Finance, Tencent quote, Longbridge OpenAPI, and Alpha Vantage-compatible implementations, plus comma-configured fallback chains, provider-level throttling/cooldowns, and Tencent/Longbridge batch quotes.
- `decision`: explicit investment decision events.
- `decision_delta`: actual legs and baseline shadow legs for quantifiable decisions, with daily/manual snapshots, stale fallback, reviews, and candidate adoption.
- `profile`: rule-driven XP, levels, attributes, badges, and bias signals.
- `ai`: provider trait with mock, OpenAI-compatible, and CLI-backed implementations.
- `settings`: runtime AI provider configuration persisted to the shared `.env` in the original repository working tree.

## Local-First Defaults

Local `.env`, the default SQLite database, attachment originals, company Markdown projections, and custom Capability manifests are read from the original repository working tree behind the Git common dir by default, so different git worktrees read and write the same configuration, investment data, and personal analytical methods. `PRUDENTIA_LOCAL_DIR` can override this directory; relative SQLite URLs resolve from that local state directory. Attachments and company projections live under `data/workspace`, capability manifests live under `capabilities/`, and the database stores only attachment/projection relative paths.

SQLite is the first persistence layer. There is no login, multi-user authorization, or broker API sync in v1. Portfolio quantity and average cost come from import/manual updates; automatic updates only refresh prices and derived values. Messages, sources, confirmation actions, company versions, trades, rule graphs, and portfolio snapshots are retained as long-lived investment facts. Active-run `message.delta` rows exist only for reconnect recovery and are deleted after the terminal full message is persisted; research cache rows are pruned after their 24-hour lifetime during a later lookup. Conversation run creation uses `BEGIN IMMEDIATE` to reserve the writer before reading thread state, preventing an unresolvable deferred-transaction upgrade when a background writer such as symbol refresh is active. Released SQLite pages remain available for future writes, and the application does not run `VACUUM` after each turn, avoiding full-database rewrites and unnecessary disk churn.

Portfolio Performance uses portfolio value snapshots and system-recorded trade adjustments. Import confirmation, draft confirmation, position edits, and position deletes write a `buy` or `sell` adjustment into `portfolio_cash_flows` when the CNY portfolio value changes; daily quote refreshes only write snapshots and never create trade adjustments. Portfolio return compounds each snapshot interval as `(ending value - interval net trade adjustment) / starting value - 1` to produce time-weighted return while also exposing the unadjusted snapshot return for explanation. Deleting down to an empty portfolio becomes the new starting boundary for later performance reads, so cleanup followed by reimport is not miscounted as investment return. The holdings table period return reuses the same `this month` / `this year` / `since inception` selector and computes each holding from its CNY value snapshot change. S&P ETF proxy, Hang Seng ETF proxy, and official SSE Composite snapshots are written only during the holding price-refresh cycle for same-period return comparison. Holding unrealized P/L uses the common broker holdings-page convention: native market value is `last_price × quantity`, native unrealized P/L is `(last_price - average_cost) × quantity`, current return rate is `unrealized_pnl / (average_cost × quantity)`, and CNY totals convert by FX.

Conversation home treats the thread as the default interaction object. Desktop uses thread, primary conversation, and context columns with the conversation taking most space; mobile moves threads and context into left/right drawers. The right panel provides current holdings, the current company view, and context used for the latest response. Send and stop share one control, no empty assistant bubble is created before text exists, and the independent runtime component only displays persisted concrete phase/activity, task level, actual model, routing reason, provider, source count, and elapsed time, including every parallel skill/agent. Company work additionally shows a deterministic complete execution plan instead of treating an agent's `(current turn/turn limit)` as the research plan: `run.plan.created` and `run.plan.step` drive scope, default-template dimensions, and every step. Active-run REST snapshots reconstruct `execution_plan` from replayed events and include unfinished capabilities; WebSocket replay drains the persisted cursor in batches, and the frontend merges events that arrive before REST hydration. Completed structured capability artifacts render independently beside the assistant message, so refresh neither infers activity nor reruns analysis.

Raw messages plus `conversation_runs` and `conversation_run_events` are the facts of record. `ConversationEngine` atomically persists user input and the run before subject resolution, deterministic classification, context assembly, optional research, tiered model invocation, structured extraction, and persistence; response generation and action projection/persistence are separate orchestration functions. The persisted thread subject and effective turn subject are independent: explicitly naming another company inside a bound company thread switches only that turn and does not rebind the thread. Context excludes the bound company's thread summary, prior turns, company view, and research cache, while turn summaries are read and written by the effective company. The dedicated `subject_resolution` module combines holdings, the bound company, security codes, and the local symbol directory into candidates and creates a company subject only for one high-confidence result. For an explicit company hint of 5-32 ASCII letters, it may also apply a bounded Damerau-Levenshtein match against same-initial directory candidates: at most one edit for short names and two for names of ten or more characters, with automatic binding only for a unique result. A lowercase pure-letter ticker becomes a strong reference only next to investment, holding, analysis, research, introduction, or company-question markers, preventing ordinary English words and greetings from becoming securities. Multiple candidates, tied fuzzy matches, or an unmatched explicit company request return clarification instead of falling back to the thread company. Clarification uses a light real-AI turn with a minimal context containing only the user's request and candidates; it performs no research or action projection and cannot update the bound thread summary. Its assistant message persists the original request and candidates so a unique code or candidate ordinal resumes the request, while a non-unique reply asks again. After identity resolves, `execution_plan` chooses a focused scope or the default comprehensive template from message intent and starts immediately without another blocking scope question. That scope/template explains what the backend will execute; it does not replace the lower-level `ResearchPlan` or Capability orchestration. Ordinary company/trade extraction always uses the standard tier rather than inheriting the slower deep model. Its timeout is 120 seconds after light/standard turns and 300 seconds after deep turns. Only an investment-system subject uses deep extraction and may propose a rule graph; other subjects' rule-graph proposals are rejected. Company response and projection context excludes historical valuation, trades, and the investment system, and company patch preparation/execution deterministically removes valuation changes. CLI and OpenAI-compatible output is parsed by asking `serde_json` for the first complete target object, so fences and trailing commentary do not corrupt extraction. WebSocket remains an `event_id` replay/subscription channel.

The conversation entry module exposes only run, thread, and confirmation-action APIs, while a dedicated runtime deep module owns subject resolution, tool orchestration, generation, and action extraction. Routing freezes a `TurnContext` containing a deterministic bounded `ToolPlan`; completed read tools then freeze a `StepContext` with the exact model input and persisted sources, so later stages do not resample mutable thread state. Strongly typed `ConversationEvent` values preserve a stable wire format. `TurnTask` propagates one cancellation signal through tools, visible generation, and action extraction, waiting for cooperative shutdown before using forced abort as a timeout fallback. Completed, failed, canceled, and interrupted updates compete through one conditional database transition, so only the first terminal writer wins.

`conversation::capabilities` is the unified capability boundary and exposes deterministic planning, execution, and registered rule-node adapters. The Capability Registry registers Rust `native` tools plus declarative `skill` and `agent` definitions by stable `id + version`, validating input/output JSON Schema, subject scope, least-privilege context, model tier, read-only side effects, confirmation policy, timeout, surface, storage policy, and initial user-visible activity. A skill is a versioned analytical method that can run directly or be preloaded into an agent. An agent is a bounded model session with its own instructions, exact skill dependencies, read-only tool allowlist, and up to eight turns. A turn executes Research first, freezes `ConversationContext`, and then runs matching Analysis/Challenge capabilities in parallel against the same snapshot; a dissent agent never reads another capability's output. On each internal turn an agent selects one exact-version allowlisted tool or submits its final Schema output. Deterministic Registry code still validates and executes every tool, allows at most four non-duplicate calls, and records bounded turn/tool/source metadata in `agent_trace`. One turn runs at most three model capabilities, each output is capped at 48 KiB, and agent observations are capped at 96 KiB. Research failures enter external-verification degradation; model-capability failures become independent failed artifacts and do not contaminate research state. Raw provider/CLI errors remain server-side, while durable events and artifacts carry stable public failure categories.

Built-in `research_company` and `research_community_insights` continue to share structured `ResearchPlan` and source validation with separate cache scopes. They expose only one bounded `focus` field to an agent; a Rust planner still produces the actual retrieval plan. Pure community requests execute only the community category, while requests that also require business-model, financial, moat, or broad-company judgment keep the complete company-research plan. Built-in `analyze_business_model` and `audit_moat` are reusable methods. `analyze_company` and `challenge_company_thesis` preload those methods and select company or community research according to the remaining evidence gap. Custom skills and agents load at startup from `capabilities/**/*.json` under the shared local root behind the Git common dir. Unknown Schema keywords, unresolved dependencies, write tools, symlinks, and unbounded configuration are rejected, while one bad file does not suppress valid neighbors. The application exposes no arbitrary `load_skill(path)`, `spawn_agent(prompt)`, Shell, browser, or plugin passthrough. The CLI provider returns structured decisions and never owns tool lifetime. See [Conversation Capabilities](capabilities.en.md) for the complete contract.

A model capability prompt has three layers: a schema-aware runtime protocol, manifest method instructions, and the output schema. The four built-in analyses have distinct domain categories rather than one generic template. Their shared finding contract records evidence sufficiency, claim type, sources and dates, causal chain, counterargument, unknowns, leading indicators, falsification, and decision impact. A model may cite only exact URLs from the frozen context or agent tool observations. Runtime code recursively validates `source_urls` and rejects an uncited `fact`. Community sources remain unverified signals. When retrieval fails and evidence is insufficient, the agent stops with a concise abstention, decisive gaps, and at most two hypotheses instead of filling gaps from model memory or walking every category. Legacy string evidence is converted only by the frontend compatibility layer; stored messages remain unchanged.

Skills and agents declaring the `rule_graph` surface and explicitly granting `rule_graph_input` register in `RuleNodeAdapterRegistry` as `id@version`. Graph activation and evaluation use that same registry, validate node/adapter kind, capability arguments, acyclicity, and node Schema, then pin the `manifest_hash` into the active graph version; editing a manifest in place without a version bump makes evaluation fail. A graph is limited to 64 nodes, 256 edges, 8 model-backed nodes, and 512 KiB, has a 600-second total deadline, and retains the latest 500 executions. Input is stored once and node traces reference it instead of duplicating context. Models cannot perform direct data writes; company-view, trade, and rule-graph changes remain confirmation cards executed by deterministic domain code.

`conversation::research::cache` owns cache reads, hashing, and expiry pruning for both ordinary company research and community-specific research; providers and tool adapters never operate on the cache table directly.

A community-specific plan preserves the confirmed company and structured request intent but does not inherit the broad-analysis default five-year financial scope, preventing unrelated filing terms from entering community queries.

Every visible reply comes from the configured real provider. Broad company analysis maps users/customers/payers, the value chain, and money flows before rating competitive intensity, relative competitive position, and durable-profit difficulty, then covers moat, default five-year financial validation, operating risks, and company quality. Response context for a company subject excludes portfolio summaries, positions, trade history, and rule graphs. Share prices, quotes, market capitalization, valuation multiples, price targets, stock returns, technical charts, ratings, personal P/L, and trading implications are ignored even when present in history or sources. Product markets, industry structure, competitors, and market share remain valid operating evidence. Visible responses have hard limits of 90, 240, and 600 seconds for light, standard, and deep tasks respectively. Action projection is capped at 120 seconds after light/standard turns and 300 seconds after deep turns. External research uses a keyless public-sources adapter by default, optional Tavily, and a disabled adapter. The default adapter retrieves SEC company filings, Yahoo Finance company-operating news, and TradingView operating viewpoints with hot/engagement data while filtering results that only discuss stock valuation, prices, or trading technique. Response context retains up to three sources per tier, with up to 8,000 characters per primary source and 2,500 per independent/community source; persistence remains bounded at nine sources per turn and 8,000 characters per source. Research execution is scheduled independently from persisted progress writes, and canceling the parent run aborts its research task to avoid pool waits and detached retrieval. Complete and partial results are cached for 24 hours; unavailable search is stated explicitly.

Moat questions are deep tasks with a dedicated response structure. Risk, competition, counterevidence, and "how to breach it" are dimensions inside that audit; the broad-company structure is selected only when the request also asks for an independent axis such as the business model, filings, or valuation. The system first rejects the inference that market share, product quality, management, execution, founder talent, marketing/distribution, or temporary technology leadership is itself a moat, then validates candidates through the chain structural mechanism -> customer or cost behavior -> competitor constraint -> retained excess returns. Broad company analysis runs the same audit. Focused moat work creates an auditable judgment card for every mechanism with sourced facts, competitor comparison, maintenance cost, counterevidence, confidence, and adjacent-verdict conditions; 1-5 scores are forbidden without observable anchors. Breach paths are ranked by probability times impact and only the top three to five are expanded. Every path repeats the same fields for attack steps, required resources, company response, economic-profit transmission, leading indicators, disconfirming conditions, missing evidence, and verdict-change point; when space is constrained, the response uses fewer paths instead of truncating the final one. Final output and `company_view_patch.moat` preserve those grounds plus strength, temporary/medium-term/structural horizon, and kill conditions. Projection may use up to 3,500 characters for `moat` and 9,000 across one company-view patch, compacting repeated prose first. User-provided material informs the method only; unverified company-specific opinions in that material are not injected into company views.

Business-model questions are deep tasks with a dedicated structure separate from broad company analysis. Broad analysis reserves at least 60% of substantive content for the business model, while focused requests reserve at least 75%. The structure covers offering/customer job, transaction lifecycle and risk ownership, monetization/profit pool, segment-level unit economics, competition/attacker economics, positive compounding conditions, and inverse failure plus multidisciplinary synthesis. Materially different products, geographies, and fulfillment models stay separate; consolidated growth is not a substitute for segment economics, and each section labels fact, inference, and unknown. It then selects three to five causally relevant lenses from microeconomics/industrial organization, accounting and corporate finance, psychology/incentives, systems thinking, game theory, technology/operations, organization/regulation, and outside-view base rates. Each lens must produce model -> observed mechanism -> required evidence -> implication, preventing vocabulary-only output. The integrated verdict distinguishes value creation from value transfer, rates the model fragile/mixed/robust, and records earliest break signals, disconfirming evidence, and kill conditions through the `business_quality` projection.

Broad company, business-model, and moat response structures are centralized in `ai::prompt::company_analysis` and append a shared investor questionnaire plus a Buffett/Munger operating audit. A knowability gate precedes every long-range scenario: it names three to five decisive variables, tests historical stability, industry/technology change, cyclicality, regulation, management dependence, and evidence quality, then classifies the enterprise as predictable, partially predictable, or not predictably bounded. A normalized base is only a starting point. Numerical five/ten-year company-operating downside/base/upside profit or owner-earnings ranges are allowed only when variables can be bounded; otherwise the projection retains qualitative scenario architecture, blockers, and evidence needed to quantify. The shared audit separately covers owner earnings and diluted per-share economics, maintenance versus growth capex, dilution, incremental ROIC, return on retained earnings, reinvestment runway and great/good/gruesome classification; management ability/integrity/candor, succession, stakeholder incentives and historical capital allocation; and debt, liquidity, contingencies, and permanent-impairment paths. Replies still end with the six-row facts/optimistic/pessimistic/verdict/missing-evidence matrix, and operating cases never use stock-market conditions, share prices, or valuation multiples.

Research planning applies only to subjects resolved as `company`. A dedicated planner reduces raw text to a `ResearchPlan` containing only company identity, structured intent, normalized annual scope, and operating-evidence queries; providers never receive positions or other private context, and no valuation or quote query is generated. Tavily uses compact bounded category-specific queries, and every source snippet is capped at 8,000 characters before cache/storage. The default public-sources adapter emits SEC capex, share compensation, diluted shares, a free-cash-flow proxy and its per-share proxy with structured units, while 10-K/20-F extraction covers owner economics, stewardship, and resilience within the 4,500-character excerpt cap. Raw XBRL JSON is never stored.

The model proposes changes but never writes investment data directly. Company-section patches, trade records, and rule-graph patches become independent confirmation cards. After user edit/confirmation, deterministic domain code executes them with target-version checks and idempotent state. A company confirmation creates an immutable version and Markdown projection. A trade confirmation uses historical FX and the import baseline to update quantity, average cost, and TWR cash flow, while corrections use reversal/replacement events. A rule confirmation creates and activates a new DAG version only after node input/output Schema, acyclicity, and exact-version adapter validation.

Security code matching uses the local `security_symbols` directory. The default public provider first reads the normalized in-repo inventory file `data/symbol-directory/public/symbols.json` and imports it into SQLite. That file is generated from the no-account public directories declared in `config/symbol-directory-public.json`; current coverage includes SSE stock/fund data, HKEX English/Traditional Chinese securities lists, and Nasdaq Trader US lists. Inventory security records only store `symbol`, `name`, `market`, and `currency`; SQLite `security_symbols` only adds the file-level `updated_at` and no longer stores provider, exchange, or asset type. Traditional Chinese security names are cleaned to Simplified Chinese before the inventory is written. Startup checks the inventory `updated_at`; the file is reused for 24 hours by default, and public sources are refreshed asynchronously only after expiry before replacing the inventory. Refresh failures only emit warnings and do not block startup or existing local matching. Import confirmation and screenshot recognition only query the local directory instead of making live fuzzy-search requests, avoiding provider rate limits and silent guessing. Chinese matching folds Traditional and Simplified variants; authorized sources such as Tushare or broker OpenAPIs can be added as future `SymbolDirectoryProvider` implementations to improve alias and Chinese-name coverage.

Decision Delta v1 does not build an unbounded world tree. Each quantifiable decision creates one actual/baseline fork, then snapshots preserve how that fork performs on later dates. The timeline summary is the sum of latest `actual_value - baseline_value` snapshots in the current filter scope, not a full counterfactual portfolio NAV.

## Conversation Safety Boundaries

Ordinary punctuation and instructions such as “update the company view” cannot create a company hint. Company research accepts only an explicit company-analysis structure or one high-confidence candidate, and ambiguous identity must be confirmed first. A pending clarification accepts a candidate ordinal, security code, short company name, or an explicit affirmation after a unique candidate. If deterministic clarification initially found no candidate, a recently canceled AI confirmation question may resume the original request only when it uniquely names one company in the local symbol directory. A generic affirmation cannot select among multiple candidates, and a new complete request supersedes the clarification. When the user explicitly restricts a turn to existing local context, the planner creates no external retrieval and a short question uses the standard model. Codex CLI action projection uses its native JSON Schema to enforce the `summary` / `actions` envelope, then decodes and validates every payload before deterministic domain code sees it. Malformed output creates a durable run warning and cannot directly mutate a company view, trade, or investment rule.

## Extension Points

- Add broker sync by introducing a `BrokerProvider` module that writes normalized transaction events.
- Add richer AI workflows by expanding `AiProvider` with memo critique, decision review, and profile narration.
- Add more market data providers behind the existing `MarketDataProvider` trait.
- Add more `SymbolDirectoryProvider` implementations behind the local security-symbol directory, such as Tushare, Choice, Futu OpenAPI, or other authorized sources.
- Add additional AI providers behind the existing `AiProvider` trait. CLI-backed providers share a reusable runner and per-tool backend enum; the current `codex` backend is intentionally implemented through `codex exec` so Codex device-code authentication remains owned by the Codex CLI.
